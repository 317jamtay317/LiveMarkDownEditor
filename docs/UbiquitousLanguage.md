# Ubiquitous Language

The shared vocabulary of the LiveMarkDownEditor domain. These terms are **authoritative**: every
type, method, namespace, variable, and test name must use them exactly. Do not introduce synonyms.
To add a concept, define it here first, then use it in code.

> Status: living document. Add terms as the domain grows.

## The product in one sentence

LiveMarkDownEditor is a **single-pane WYSIWYG** Markdown editor: the user reads and edits
formatted content directly (a heading looks like a heading), never raw Markdown syntax, while the
file on disk stays plain Markdown and updates live when changed by anyone — another user or an AI.

## Terms

| Term | Definition |
| --- | --- |
| **Markdown Document** | The source text authored in Markdown syntax. The primary aggregate of the editor and the single canonical representation of the content. Its source text is what is persisted to the Watched File. |
| **Visual Document** | The formatted, editable projection of a Markdown Document that the user actually sees and edits in the single WYSIWYG pane. It shows rendered formatting (bold as bold, a heading as a heading) and never exposes raw Markdown syntax. In the WPF UI it is realised as a `FlowDocument`. |
| **Project** | The act of transforming a Markdown Document's source text into a Visual Document for display and editing. (Source → Visual.) |
| **Capture** | The act of transforming the user-edited Visual Document back into Markdown Document source text. (Visual → Source.) The inverse direction of Project. |
| **Round-Trip** | A Project immediately followed by a Capture. A Round-Trip must preserve the semantic content of the Markdown Document (see Invariants). |
| **Canonical Markdown** | The normalised source text Capture emits — the fixed point of a Round-Trip (INV-005). Markdown authored in a style Project and Capture do not preserve (setext rather than ATX headings, `_` rather than `*` emphasis, differing blank-line spacing) Round-Trips to the same Canonical Markdown as the same content authored canonically. Round-Tripping source text is therefore how its Canonical Markdown is obtained. |
| **Render** | The act of transforming a Markdown Document's source text into its HTML representation. Used for export and interoperability, **not** for the on-screen editing surface (that is the Visual Document). |
| **Rendered Output** | The HTML produced by rendering a Markdown Document. |
| **Export as HTML** | The action that writes the Active Session's Rendered Output to a file the user chooses, in the Export Shape they choose alongside it. It is what makes Rendered Output reachable: Render exists for export and interoperability, and an export is how that HTML leaves the editor. Exporting reads the Markdown Document and writes a *different* file — never the Watched File — so it is not an edit (INV-032). |
| **Export Shape** | Which of two forms an Export as HTML writes. A **Standalone Page** is the Rendered Output wrapped in a complete HTML document — a title and an embedded stylesheet — so the file stands on its own when opened in a browser. An **HTML Fragment** is the Rendered Output alone, for pasting into a page that supplies its own surroundings. The two carry the same Rendered Output and differ only in what surrounds it. |
| **Export as PDF** | The action that writes the Active Session's Markdown Document to a PDF file the user chooses. Because a PDF cannot embed the Visual Document, its content is **re-laid-out** from the Markdown — so it need not match the Visual Document line for line. Like Export as HTML it reads the Markdown Document and writes a *different* file — never the Watched File — so it is not an edit (INV-033). |
| **Print** | The action that sends the Active Session's Visual Document to a printer, laid out across pages as it appears on screen. It prints the Visual Document itself (the `FlowDocument`), re-projected from the current source so a Folded Section's hidden Section Body prints too — Print means the whole document, never merely the visible part. Printing reads the document and writes no file the editor owns, so it is not an edit (INV-034). On Windows the print dialog's "Microsoft Print to PDF" makes Print a way to produce a PDF at full fidelity. |
| **Copy as Rich Text** | Copying the current selection to the clipboard in rich flavors so it pastes formatted wherever it lands: **RTF** (which Word and Outlook consume, produced by the editing surface itself) and **HTML** (which web editors — Gmail, Google Docs, Slack — consume, rendered from the selection). The HTML flavor is produced from the blocks the selection spans, so a copied Link or Heading is the same as a saved one. Copying reads the document and is not an edit (INV-035). |
| **Copy as Markdown** | The command that copies the current selection's Markdown source to the clipboard, for pasting into a Markdown-aware target (a repository, chat, or another editor). Like Copy as Rich Text it Captures the blocks the selection spans — a partial selection copies the whole blocks it touches — and is not an edit (INV-035). |
| **Smart Paste** | Pasting that adapts to what is on the clipboard: a URL pasted over a selection turns the selection into a Link; an image is written to a file beside the Watched File and inserted as an Image; HTML is converted to Markdown and pasted as formatted content, Preformatted HTML as a Code Block. Anything else pastes as it normally would (INV-041). |
| **Preformatted HTML** | HTML whose whitespace is its content: a `<pre>`, or any element declaring `white-space: pre` — the shape a code editor puts on the clipboard, one element per line. Smart Paste converts it to a Code Block rather than to prose, because that is the only place a snippet's indentation survives: HTML's ordinary whitespace rules collapse it, and a Markdown paragraph cannot carry leading spaces (INV-041). |
| **Workspace State** | The Workspace's persisted snapshot across runs: the open Tabs' Watched File paths (in Tab order) and the Recent Files. Only saved documents are recorded — a Tab with no Watched File, and any unsaved edits, are never persisted (INV-037). |
| **Recent Files** | The recently opened or saved Watched File paths, newest first, distinct and capped. Shown in the Open Recent menu and mirrored to the Windows Jump List, and persisted as part of the Workspace State (INV-036). |
| **Restore** | The action, at startup, of reopening the previous run's Workspace from its persisted Workspace State — its saved Tabs (by path, skipping any that have gone) and its Recent Files. An empty or unreadable state restores nothing, leaving the single empty Tab (INV-008, INV-037). |
| **Jump List** | The Windows taskbar list of Recent Files, each entry reopening that file in the editor (as a Startup Document, INV-020). |
| **Follow Link** | Ctrl+Clicking a Link in the Visual Document to open its destination — a web address in the default browser, a relative Markdown file in a new Tab (resolved against the Base Directory). Following reads the document and is not an edit (INV-038). |
| **Status Bar** | The slim bar along the bottom of the window showing the Active Session's Watched File, its Document Statistics, the caret's line and column, and the Current Section. Presentation-only — it reflects the document and never changes it (INV-039). |
| **Document Statistics** | The word count, character count, and estimated reading time of a document's prose — a deterministic function of its text, shown in the Status Bar (INV-039). |
| **Editor Session** | An active editing context holding the current Markdown Document, its Visual Document projection, the associated Watched File, and whether unsaved edits exist. |
| **Watched File** | The file on disk that backs the Editor Session. Its contents are the persisted Markdown Document source text, and it is monitored for External Change. |
| **External Change** | A modification to the Watched File made outside the Editor Session (by another user, process, or AI) while the session is open. An External Change whose new contents share the Editor Session's Canonical Markdown — the Editor Session's own save, or another writer merely restyling the file — changes bytes but no content, and is ignored (INV-026). |
| **Conflict** | The situation where an External Change is detected while the Editor Session has unsaved edits — the disk and the in-memory Markdown Document have diverged. A Conflict is never resolved by silently discarding either side; it is surfaced to the user for a decision (keep edits, reload from disk, or view the difference). |
| **Conflict Difference** | The line-based comparison between the two sides of a Conflict: the Editor Session's unsaved Markdown Document source text and the conflicting on-disk contents of the Watched File. It is an ordered list of Difference Lines accounting for every line of both sides, computed as a pure function (a longest-common-subsequence over lines). Lines are compared by content with line terminators excluded — a CRLF/LF difference, or a trailing newline alone, never makes lines differ. Computing a Conflict Difference is view-only (INV-021). |
| **Difference Line** | One line of a Conflict Difference: its text together with its **Difference Line Kind** — **Unchanged** (present on both sides; shown as context), **Session Only** (present only in the Editor Session's unsaved text), or **Disk Only** (present only in the Watched File's on-disk contents). The kinds name where a line lives, never which side should win. Within a replaced run, Session Only lines precede Disk Only lines. |
| **View Difference** | The act of showing the Active Session's Conflict Difference — an overlay covering the editing area while the Conflict bar stays visible — or hiding it again; the action toggles. Available only while a Conflict awaits resolution; resolving the Conflict (keep edits or reload from disk) hides it. View Difference is presentation-only: it never changes the Markdown Document, the Watched File, or the Conflict (INV-021). |
| **Workspace** | The set of open Editor Sessions the user is working with at once, presented as Tabs, with exactly one of them the Active Session whenever any are open. The editor's shell: it is what lets several Markdown Documents be open and edited side by side. A Workspace opens with a single empty Editor Session, but it **may become empty**: closing the last Tab leaves it with no open Editor Session and no Active Session, showing an Empty-Workspace Placeholder until the user opens or creates a document. |
| **Active Session** | The single Editor Session in the Workspace currently shown in the editing pane and targeted by the save and formatting actions. Selecting a Tab changes which Editor Session is the Active Session. There is **no** Active Session when the Workspace is empty. |
| **Empty-Workspace Placeholder** | The message shown in the editing area when the Workspace has no open Editor Session (every Tab has been closed): a prompt inviting the user to open or create a document. Presentation-only. |
| **Tab** | The visual handle representing one Editor Session within the Workspace. Selecting a Tab makes its Editor Session the Active Session; closing a Tab ends that Editor Session (never discarding unsaved edits without a decision — see Invariants). |
| **Heading** | A block of the Visual Document shown at one of six **Heading Levels** — Level 1 the most prominent, Level 6 the least — and captured as an ATX heading (`#` repeated once per level, then the text). A Heading is distinguished visually by size alone, never by weight: an inherited bold weight would make Capture read every Heading as inline-bold. It is the construct a Section is built from — a Section's Heading is its Section Heading — and it is what the Outline lists and what Folding Folds. |
| **Section** | A heading together with all Visual Document content that follows it, up to (but not including) the next heading of equal or higher level. A Section's heading is its **Section Heading**; the content it owns is its **Section Body**. Sections nest: a Section Body may contain lower-level Sections. |
| **Fold** | A view-only operation on the Visual Document that hides a Section's Section Body while leaving its Section Heading visible, the way Visual Studio collapses a method or region. The inverse is **Unfold**. Fold state is presentation-only: it is never part of the Markdown Document and never changes its source text or the result of a Capture. |
| **Collapse All / Expand All** | The two Workspace-level fold actions: **Collapse All** Folds every Section; **Expand All** Unfolds every Folded Section. Like any Fold, they are view-only and never change the Markdown Document. |
| **Editor Gutter** | The margin strip along the left edge of the WYSIWYG editing surface. It is pure presentation — it shows a **Line Number** for each visible line and a **Fold Toggle** beside each Section Heading — and is never part of the Markdown Document. |
| **Line Number** | The 1-based ordinal of a visible line, shown in a gutter. In the Editor Gutter it numbers each rendered line of the Visual Document, counting each soft-wrapped continuation as its own line the way a code editor does; the lines of a Folded (hidden) Section Body are not numbered. In the Source Gutter it numbers each source line of the Markdown Document (the Source Panel does not wrap, so one source line is one row). Line Numbers are presentation-only. |
| **Source Gutter** | The margin strip along the left edge of the Source Panel. It is pure presentation — it shows a Line Number for each visible source line of the Markdown Document — and is never part of the Markdown Document. It is the Source Panel counterpart of the Editor Gutter. |
| **Fold Toggle** | The chevron shown in the Editor Gutter beside a Section Heading — pointing down when the Section is Unfolded, right when it is Folded. Activating a Fold Toggle Folds or Unfolds that Section. |
| **Outline** | The ordered list of every Section Heading in the Active Session's Visual Document, in document order, each with its heading level and text. It lists *all* Section Headings, including those inside a Folded Section Body. The Outline is a view-only projection of the Visual Document — building it never changes the Markdown Document. |
| **Outline Entry** | One Section Heading as it appears in the Outline: a navigation target showing the heading's text, indented by its heading level so the document's structure is visible at a glance. Activating an Outline Entry Navigates to its Section Heading. An Outline Entry that has Sections nested within it (Outline Entries of lower level immediately following it) can be **Collapsed** in the Navigation Panel to hide those nested Outline Entries, or **Expanded** to show them again. This disclosure is Navigation-Panel-only: it changes neither the Markdown Document nor any Fold state in the Visual Document, and is distinct from a Fold (which hides a Section Body in the editing surface, not Outline Entries in the panel). |
| **Navigate** | The act of moving the editing surface to a Section Heading and selecting it, in response to activating its Outline Entry. If the Section Heading is inside a Folded Section Body, the enclosing Section is first Unfolded so the heading is revealed. Navigation is view-only: it selects and scrolls, but never changes the Markdown Document. |
| **Navigation Panel** | The toggleable panel along the left edge of the Workspace that presents the Active Session's Outline for Navigation. It is presentation-only — showing, hiding, or Navigating from it never changes any Markdown Document — and is hidden until the user toggles it on. It is presented as a tab of the **Side Dock**, sharing the left edge with the Folder Panel rather than taking a column of its own. |
| **Current Section** | The Section whose Section Heading most immediately precedes the caret in the Visual Document — the Section the user is currently editing within. The Navigation Panel highlights the Current Section's Outline Entry so the user can see where they are as they edit or scroll. |
| **List** | A run of one or more List Items shown in the Visual Document, each with a leading marker: a bullet for an **Unordered List** or an incrementing number for an **Ordered List**. It is captured as canonical Markdown list syntax — `- ` before each Unordered item, `1. ` (incrementing from the list's start) before each Ordered item. |
| **List Item** | A single entry of a List, holding that entry's Visual Document content (at minimum a paragraph's worth of inlines). A List Item may carry a **Task Marker** — a checkbox captured as `[ ]` (unchecked) or `[x]` (checked). |
| **Task List** | A List whose List Items carry Task Markers. A Task Marker is captured immediately after its List Item's own marker — `- [ ] todo` — and exists **only** on a List Item; there is no Task Marker outside a List. An Unordered Task List shows **no bullet**: a Task Marker's checkbox *is* its List Item's marker, and a bullet beside a checkbox is one marker too many. An Ordered Task List keeps its numbers — only the bullet is redundant beside a checkbox; the numbers still carry the items' order. A Task List **continues** across a paragraph break the way any List does: breaking the line in a task item gives the new List Item its own unchecked Task Marker, exactly as a bullet or a number carries to the next item. |
| **Link** | An inline reference to a destination URL, shown in the Visual Document as its clickable link text (never as raw `[text](url)` syntax) and captured back to that syntax. A bare URL written in text (an **autolink**) is shown and captured as the URL itself. |
| **Image** | An inline image reference, captured as `![alt](url)`. Shown in the Visual Document as the picture its **Image Source** names — the point of a WYSIWYG editor is that the picture is what the user sees. An Image whose picture cannot be shown falls back to its **alt text**, which is also what a screen reader is given for one that can (INV-031). |
| **Image Source** | The URL an Image names. It is either **absolute** — a remote `http(s)` address, or a rooted path — or **relative**, naming a file with respect to the Base Directory. A relative Image Source is what makes the Base Directory an input to Project (INV-003). |
| **Base Directory** | The folder the Editor Session's file lives in, and the folder a relative Image Source is resolved against. An unsaved Editor Session has no file and so no Base Directory: its relative Image Sources resolve against nothing and fall back to alt text, because there is no folder yet for "beside this document" to mean (INV-031). |
| **Block Quote** | A quoted passage, shown in the Visual Document with a left rule and muted text, and captured with a `> ` prefix on every line. |
| **Strikethrough** | An inline span of struck-through prose, shown with a line through it and captured as `~~text~~`. |
| **Code Block** | A block of preformatted, monospaced text shown without Markdown interpretation, captured as a fenced block (```` ``` ````) with its optional language. |
| **Code Span** | A run of inline code within prose, shown monospaced and set off from the surrounding text, captured as `` `code` ``. It is the inline counterpart of a Code Block. |
| **Code Shading** | The shaded background drawn behind every Code Block and Code Span to set code off from prose. It is presentation-only — it never changes the Markdown Document — and is drawn as a **read-only overlay** on the Visual Document (not as the code's own background), so it recolours with the active light/dark palette without re-formatting the document. |
| **Code Region** | A single span of the Visual Document that receives Code Shading: either a whole Code Block or one inline Code Span. The set of Code Regions is a view-only projection of the Visual Document — computing it never changes the Markdown Document. |
| **Thematic Break** | A horizontal rule separating content, shown as a horizontal line and captured as `---`. |
| **Table** | A grid of cells with a header row and per-column alignment, shown in the Visual Document as a real table and captured as a GFM pipe table. |
| **Source Panel** | The toggleable panel that shows the Active Session's Markdown Document as raw, editable Markdown **source text** — the plain-text counterpart to the Visual Document, shown alongside it. It is the one surface that *deliberately* exposes Markdown syntax (a heading shows as `# Title`), where the Visual Document never does. It is not a separate Render: it displays the Markdown Document's own source text. Editing the Source Panel edits that source directly, which **Projects** an updated Visual Document; editing the Visual Document **Captures** back into the source, which the Source Panel reflects. The two are always kept in sync — they are two views of the same Markdown Document (see Invariants). Like the Navigation Panel it is presentation-only chrome and is hidden until the user toggles it on. |
| **Scroll Sync** | The keeping of the Visual Document and the Source Panel aligned as the user scrolls: scrolling either view proportionally scrolls the other to the same relative position, in both directions. **Proportional** means by fraction of scrollable height — the top of one view maps to the top of the other, the bottom to the bottom, and the midpoint to the midpoint — not line-for-line. Scroll Sync is active only while both views are shown. It is presentation-only: it moves viewports only and never changes any Markdown Document. |
| **Find** | Locating every occurrence of a search query within the Active Session's Visual Document text and highlighting them, so the user can see and step between them. Find is view-only: it highlights, scrolls, and selects, but never changes the Markdown Document. Replacing a Match is a separate operation — a real edit — and is not part of Find. |
| **Match** | A single occurrence found by a Find: a contiguous span of the Visual Document's text equal to the query, compared case-insensitively. A Find yields its Matches in document order and they never overlap. |
| **Current Match** | The one Match the user is presently on — highlighted more strongly than the other Matches and scrolled into view. **Find Next** and **Find Previous** move the Current Match to the following or preceding Match, wrapping around the ends of the document. |
| **Find Bar** | The toggleable bar that presents Find: a query box, the Current Match's position within the Match count (e.g. "3 of 12"), and Find Next / Find Previous. It is hidden until the user opens it (Ctrl+F) and dismissed with Escape. It also carries the **Replace Row**, shown only when the user opens it for Replace (Ctrl+H). |
| **Replace Row** | The part of the Find Bar that presents Replace: a Replacement box, Replace, and Replace All. Hidden until the user opens the Find Bar for Replace (Ctrl+H), and divided from the Find row above it. The Find Bar's own chrome is presentation-only, but — unlike the rest of it — the Replace Row's two commands are real edits. |
| **Watermark** | The hint a text box shows in place of its text while it is empty — "Find" in the Find Bar's query box, "Replace" in the Replace Row's Replacement box — so a borderless, unfocused, empty box still says what it is for. Presentation-only: a Watermark is never part of the box's text, and so can never reach the Markdown Document. |
| **Replacement** | The text a Match is swapped for. It is inserted verbatim, exactly as typed: a Match is found case-insensitively, but the Replacement never adapts its case to the Match it replaces. It inherits the Match's formatting when the Match *has* one — replacing a word inside bold text leaves it bold — but a Match spanning a formatting boundary has no single formatting to inherit, so its Replacement is plain. An empty Replacement deletes the Match. |
| **Replace** | The edit that swaps the Current Match for the Replacement, then moves the Current Match to the next Match. Unlike Find, Replace is a real edit: it changes the Visual Document, which Captures back into the Markdown Document. |
| **Replace All** | The edit that swaps every Match in the Markdown Document for the Replacement, in a single step that one undo restores. It Unfolds every Folded Section first, so it never misses an occurrence hidden inside a Folded Section Body — "All" means the whole Markdown Document, not merely the visible part of it. It replaces exactly the Matches present when it is invoked, so a Replacement that itself contains the query is not replaced again. |
| **Spell Check** | The judging of the Visual Document's prose against a Dictionary and the marking of each Misspelling. It is camelCase- and punctuation-aware: a code-like identifier is segmented into its sub-words and each is judged on its own, and runs marked as code are skipped entirely. Spell Check is presentation-only — marking a Misspelling never changes the Markdown Document. |
| **Command Bar** | The single strip of actions along the top of the Workspace, above the Tabs: the file actions (New, Open, Open Folder, Save, and the Export Menu), the Formatting Actions (Bold, Italic, the Heading Level Picker, the List Picker, and the Insert Menu's link, image, table, block-quote and Flowchart Builder actions among them), and the view controls (the fold actions and the Folder Panel, Navigation Panel, Source Panel and Diagram Preview toggles, gathered as the View Menu), with the theme toggle pinned at the right. It is the one place every action is reachable by mouse — the counterpart of the key gestures — and it is chrome: the bar itself never changes a Markdown Document, though the actions on it may. Each action is shown as a Command Icon rather than a word; the Insert Menu and View Menu groups show their individual icons while they fit and collapse to a single dropdown only when the row would run off-screen (INV-054). |
| **Command Icon** | The glyph naming one Command Bar action, shown in place of a word. A Command Icon is drawn (a vector outline), never typed as a text character, so that it stays sharp at any scale and takes its colour from the active palette like everything else. It is presentation-only: it names an action, it never performs one. Because an icon shows no word, every Command Icon is accompanied by a **Name** — read aloud by a screen reader and shown, together with what the action does and its key gesture, in the action's **Command Tip** on hover — so that the action is never identified by its picture alone. |
| **Command Tip** | The themed tooltip shown on hovering a command control — a Command Bar action, one of its dropdowns, an entry inside a dropdown, or any of the editor's command buttons. It names the action and explains it, in place of a bare one-line hint: the action's **Name** (the same Name its Command Icon carries) as a heading, a short line saying what the action *does*, and — when the action has one — its key gesture, shown as a small chip. It is shown whether the action is available or not: a **disabled** command still shows its Command Tip on hover, so the tip can say what the action is and, by extension, why it is currently unavailable. A Command Tip is presentation-only: it names and explains an action, it never performs one, and — like every other surface — it follows the active light/dark theme (it is drawn from the palette, never in WPF's stock system-drawn tooltip chrome). |
| **Formatting Action** | An edit applied to the Active Session's Visual Document through the Command Bar or a key gesture — bold, italic, Toggle Strikethrough, Toggle Code, Set Heading Level, Insert Link, Insert Image, Toggle Block Quote, Insert Table, Add Row, Add Column, Remove Row, Remove Column, Toggle Unordered List, Toggle Ordered List, Toggle Task List. Unlike the view-only operations (Fold, Find, Code Shading, …), a Formatting Action is a real edit: it changes the Visual Document, which Captures back into the Markdown Document. A Formatting Action always reaches the document through the Command Bar or a key gesture: an edit made by clicking the Visual Document directly (Toggle Task Marker) is equally a real edit, but is **not** a Formatting Action. |
| **Insert Link** | The Formatting Action (Ctrl+K) that turns the selection into a Link to a destination URL, or inserts a new Link at the caret. It asks for the Link's text and URL through the Link Prompt, seeded with the selection as the proposed text. A Link is nothing without a destination, so an empty URL inserts no Link (INV-030). |
| **Insert Image** | The Formatting Action that inserts an Image at the caret, asking for its alt text and source URL through the Link Prompt. It is the Image counterpart of Insert Link, and is bound by the same rules (INV-030). |
| **Link Prompt** | The dialog that asks the user for a Link's text and destination URL, or an Image's alt text and source URL. It is the one place a URL is typed: the Visual Document never shows raw `[text](url)` syntax, so a Link's destination has nowhere else to be edited. Dismissing it makes no edit at all (INV-030). |
| **Toggle Strikethrough** | The Formatting Action that strikes the selection through, or restores struck-through prose to plain text. It is symmetric over where the Strikethrough came from: a Strikethrough the Projector loaded and one a previous toggle applied are removed alike (INV-029). |
| **Toggle Block Quote** | The Formatting Action that turns the blocks the selection touches into a Block Quote, or turns the selected Block Quote's blocks back into plain blocks. It always quotes **whole blocks**: a `> ` prefix applies to a line, so quoting part of a paragraph is not expressible in Markdown (INV-028). |
| **Set Heading Level** | The Formatting Action that makes the block at the caret a Heading of a chosen Heading Level (1–6), or — choosing **Paragraph** — turns a Heading back into a plain paragraph. It sets a level rather than toggling one: choosing the level a Heading already has leaves it unchanged, so the action is idempotent and only Paragraph clears a Heading. Because the action never toggles, Paragraph is the *only* way back to plain text, so it carries its own key gesture (Ctrl+0) beside the levels' (Ctrl+1–Ctrl+6). It changes a block's level, never its content: the text and inline formatting of the block it is applied to survive the change (INV-027). |
| **Heading Level Picker** | The Command Bar dropdown that presents Set Heading Level: one Command Icon opening the seven choices — Paragraph, then Heading 1 through Heading 6. Like the Table dropdown, its entries keep their words: a dropdown is a list of choices to read, and six near-identical glyphs would separate the levels far less than the words do. **Paragraph leads the list**, divided from the levels below it: it is the way back to plain text, and the action does not toggle, so a Heading's only exit is this entry — last in the list it read as an afterthought and was missed. |
| **List Picker** | The Command Bar dropdown that presents the three List Formatting Actions — Toggle Unordered List, Toggle Ordered List, and Toggle Task List — under one Command Icon, the way the Heading Level Picker presents Set Heading Level. Each entry pairs its word (Bulleted list, Numbered list, Checklist) with the action's own Command Icon. This is the deliberate exception to the words-only dropdown rule: unlike the Heading levels or the Table actions — whose glyphs are near-identical, so their entries stay words alone — a bullet, a number, and a checklist read apart at a glance, so the icon earns its place beside the word. |
| **Export Menu** | The Command Bar dropdown that gathers the ways a finished document produces output — Export as HTML, Export as PDF, and Print — under one Command Icon, each entry pairing its word with the action's own distinct Command Icon (the export tray, the PDF page, the printer) — a discriminating glyph beside the word, as in the List Picker. The grouping is presentational: each action keeps its own meaning and invariant, and Print is not an Export (INV-034). |
| **Insert Menu** | The Command Bar group for inserting content — Insert Link, Insert Image, Insert Table (with Add Row, Add Column, Remove Row and Remove Column), the Flowchart Builder, and Toggle Block Quote. Unlike the always-a-dropdown Export Menu, it is **adaptive**: while the Command Bar is wide enough it shows these as individual Command Icons, and it collapses to a single dropdown gathering the same actions only when the row would otherwise run off-screen, expanding again when there is room (INV-054). |
| **View Menu** | The Command Bar group for the view controls — the Folder Panel, Navigation Panel, Source Panel and Diagram Preview toggles (each shown checked while its panel is open) and the fold-all actions (Collapse all, Expand all). Like the Insert Menu it shows its individual Command Icons while they fit and collapses to a single dropdown only when the row would overflow — collapsing before the Insert Menu, and returning to icons the moment the window grows (INV-054). |
| **Toggle Code** | The Formatting Action that turns the selection into code, or code back into prose. A selection within a single line becomes a Code Span; a selection spanning multiple whole lines, or covering a whole line, becomes a Code Block. A Code Block is made only out of top-level paragraphs, because that is all a fence can replace: within a List, a Block Quote, or a Table the action makes a Code Span instead — one per line it touches — rather than swallowing the whole structure (INV-018). Applied where the selection or caret already sits inside a Code Span or Code Block, it removes that code formatting instead. |
| **Insert Table** | The Formatting Action that inserts a new Table — three columns, a header row, and two empty body rows, with no explicit column alignment — at the caret. Available only while the caret is **not** inside a Table (a Table is never nested inside another Table). |
| **Add Row** | The Formatting Action that inserts a new empty row into the Table containing the caret, immediately below the caret's row. Available only while the caret is inside a Table. |
| **Add Column** | The Formatting Action that inserts a new empty column into the Table containing the caret, immediately to the right of the caret's column. The new column extends every row — header and body — so the Table stays rectangular. Available only while the caret is inside a Table. |
| **Remove Row** | The Formatting Action that deletes the caret's row from the Table containing the caret, and moves the caret into the row that takes its place. It is the counterpart of Add Row. A pipe table is nothing without its header, so Remove Row is available only while the caret is inside a Table **and not in its header row** — the header is removed by deleting the whole Table, never by shrinking it (INV-019). |
| **Remove Column** | The Formatting Action that deletes the caret's column from the Table containing the caret — its cell in every row, header and body, and its column alignment with it — so the Table stays rectangular. It is the counterpart of Add Column. Available only while the caret is inside a Table **with more than one column**: a Table with no columns is not a Table (INV-019). |
| **Block Island** | A block the caret cannot type past from within: a Table and a Mermaid Diagram. A Mermaid Diagram holds a picture rather than text, so a caret inside it has nowhere to go; a Table's cells hold text, but there is no line after the Table itself. A Block Island is therefore never the last block of a Visual Document — an empty paragraph always follows one, so the user can always carry on typing below it (INV-055). |
| **Toggle Unordered List** | The Formatting Action that turns the paragraphs in the selection into an Unordered List, or turns the selected Unordered List back into plain paragraphs. Applied to an Ordered List it converts that List to an Unordered one rather than removing it — the two List kinds toggle between each other, never off through one another. |
| **Toggle Ordered List** | The Formatting Action that turns the paragraphs in the selection into an Ordered List, or turns the selected Ordered List back into plain paragraphs. Applied to an Unordered List it converts that List to an Ordered one rather than removing it — the counterpart of Toggle Unordered List. |
| **Toggle Task List** | The Formatting Action that gives every List Item in the selection an unchecked Task Marker, or — when every one of them already carries a Task Marker — removes them all, making it a Task List or a plain List. Applied outside a List it first makes the selected paragraphs an Unordered List and then marks them: a Task Marker exists only on a List Item, so the action **makes** that List rather than refusing until the user has made one themselves. |
| **Toggle Task Marker** | The edit that flips one Task Marker between unchecked (`[ ]`) and checked (`[x]`), made by clicking that Task Marker's checkbox in the Visual Document. It reaches the document by direct manipulation rather than through the Command Bar or a key gesture, so it is **not** a Formatting Action — but it is equally a real edit: it changes the Visual Document, which Captures back into the Markdown Document. It changes that one Task Marker's state and nothing else. |
| **Startup Document** | A Markdown file path handed to the editor when it is launched — on the command line, or by the operating system shell when the user opens an `.md` file with the editor ("Open with", double-click on an associated file). The editor opens it into the Workspace at startup as a Watched File, exactly as if the user had opened it through the file picker. |
| **Single Instance** | The rule that at most one editor process serves the user at a time. Launching the editor while it is already running forwards the Startup Document (if any) to the running Workspace — which opens it, or activates its existing Tab (INV-009) — and brings the running window to the front, instead of starting a second editor. |
| **Misspelling** | A single word (or sub-word of a camelCase identifier) the Dictionary rejects, marked in the Visual Document with a red squiggle under exactly its span. |
| **Spelling Suggestion** | A correctly-spelled word the Dictionary offers as a replacement for a Misspelling. The user reaches a Misspelling's Spelling Suggestions by right-clicking it, on the Editor Context Menu; choosing one replaces the Misspelling's span with the Suggestion, which (like any edit to the Visual Document) Captures back into the Markdown Document. A Misspelling may have no Spelling Suggestions. |
| **Editor Context Menu** | The editing surface's right-click menu: a Misspelling's Spelling Suggestions and Add to Dictionary when the pointer is over one, then the clipboard commands (Cut, Copy, Copy as Markdown, Paste). It is refilled on every opening so it reflects the word actually under the pointer, and it is always the editor's own menu rather than WPF's built-in text-editor menu — which knows nothing of this editor's Spell Check and so could only ever show a right-click with no Suggestions on it (INV-057). |
| **Dictionary** | The authority that decides whether a word is a Misspelling and offers Spelling Suggestions for it. It is the operating system's installed speller together with the User Dictionary: a word the user has accepted is never a Misspelling, whatever the speller thinks (INV-040). When the speller is unavailable, every word is treated as correctly spelled so Spell Check simply shows nothing rather than failing. |
| **User Dictionary** | The words the user has accepted through Add to Dictionary, which the Dictionary treats as correctly spelled. It is persisted, so an accepted word holds across runs (INV-040). |
| **Add to Dictionary** | The action, offered on a Misspelling's context menu beside its Spelling Suggestions, that accepts the word into the User Dictionary. The word stops being marked a Misspelling — permanently, because the User Dictionary is persisted (INV-040). |
| **Folder Workspace** | A root folder the user has opened to browse its Markdown Documents as a tree, turning the editor into a lightweight knowledge base. It is navigation chrome like the Navigation Panel: opening a Folder Workspace, browsing it, and Expanding or Collapsing its folders never change any Markdown Document (INV-043). Activating one of its files opens it in a Tab exactly as the file picker would — an already-open file activates its existing Tab rather than duplicating (INV-009). It is **distinct from the Workspace**: the Workspace is the set of open Editor Sessions shown as Tabs, whereas a Folder Workspace is a folder on disk being browsed. The open Folder Workspace is persisted and Restored across runs as part of the Workspace State (INV-045), and it tracks its root live — a Markdown Document added, removed, or renamed under it updates the Folder Tree (INV-044). |
| **Folder Tree** | The tree of Markdown Documents a Folder Workspace presents: only Markdown files appear, any folder that contains no Markdown Document beneath it (directly or transitively) is pruned, and within every folder the child folders precede the files, each ordered case-insensitively by name. It is a pure, deterministic projection of the root folder and the set of files beneath it — the same inputs always produce the same Folder Tree (INV-042). |
| **Folder Entry** | One node of a Folder Tree: either a **Folder** — a branch that holds child Folder Entries and can be Expanded or Collapsed — or a **File** — a leaf naming a Markdown Document that opens as a Watched File in a Tab when activated (INV-043). Which of the two it is, is its **Folder Entry Kind**. A Folder Entry of kind File has no children. |
| **Folder Panel** | The toggleable panel along the left edge of the Workspace that presents a Folder Workspace's Folder Tree for browsing. Like the Navigation Panel and the Source Panel it is presentation-only chrome — showing, hiding, browsing, or Expanding and Collapsing folders within it never changes any Markdown Document (INV-043) — and it is hidden until the user opens a Folder Workspace. Unlike the Navigation Panel it is **not** tied to the Active Session: it presents the opened folder even when no Tab is open, so the user can browse to a document and open it. Along with the Navigation Panel it is presented as a tab of the **Side Dock**. |
| **Side Dock** | The tabbed container along the left edge of the Workspace that hosts the Folder Panel and the Navigation Panel as tabs, so the two navigation panels need not each take a column of their own. It shows a tab for every one of those panels that is toggled on and, of the shown tabs, the content of the single **Selected** one; the Side Dock itself is shown only while at least one of its panels is toggled on, and hidden when neither is. Selecting a tab, toggling a panel on or off, and showing or hiding the Side Dock are presentation-only — none changes any Markdown Document, Editor Session, Folder Tree, or Fold state (INV-046). The Source Panel is **not** docked here: it is an editing surface shown beside the Visual Document, not navigation chrome. |
| **Mermaid Diagram** | A fenced Code Block whose language (info string) is `mermaid`, whose content is Mermaid diagram source — a flowchart, sequence, class, state, gantt, pie, ER, or any of the other diagram kinds Mermaid defines. In the Visual Document it is shown as its **rendered picture**, produced from its source by a Mermaid renderer in the active light/dark theme and re-rendered when the theme changes (INV-047); a diagram the renderer cannot produce falls back to showing its source text, so it never leaves a hole. Its source is not edited inline — it is edited through the **Flowchart Builder** (by double-clicking the picture) or the **Source Panel** — and it is Captured as exactly the fenced Code Block it is, so a Mermaid Diagram is a Code Block the editor renders specially, not a new block kind (INV-004, INV-047). |
| **Diagram Preview** | The rendered picture of the **selected** Mermaid Diagram, shown larger in the Preview Panel and re-rendered as the diagram's source or the selection changes. It is produced from the diagram's source by a Mermaid renderer (Mermaid.js) — the same picture the Visual Document shows inline (INV-047), shown here in its own pane. Presentation-only: rendering a Diagram Preview never changes the Markdown Document (INV-047). When no Mermaid Diagram is selected there is no Diagram Preview, and the Preview Panel says so. |
| **Preview Panel** | The toggleable pane along the right edge of the Workspace, beside the Visual Document, that hosts the Diagram Preview. Like the Source Panel it is an output view shown beside the editing surface — not navigation chrome, so it is **not** a tab of the Side Dock — and it is hidden until the user toggles it on. It follows the active light/dark theme. Presentation-only: showing, hiding, or rendering in it never changes any Markdown Document (INV-048). |
| **Panel Column** | The column of the Workspace a toggleable side panel — the Source Panel or the Preview Panel — occupies. While its panel is shown it is as wide as the user last dragged it, and while its panel is hidden it takes **no width at all**, so the Visual Document fills the whole Workspace: a hidden panel leaves no trace of itself behind (INV-056). Its width is view state — resizing or toggling a Panel Column never changes any Markdown Document (INV-014, INV-048) — and it is never narrower than the panel needs to stay usable. The Side Dock is not a Panel Column: it is a fixed-width dock, not a splittable pane. |
| **Panel Splitter** | The draggable divider between the Visual Document and a Panel Column, by which the user sets that panel's width. Dragging one is presentation-only, is remembered for as long as the app runs — hiding the panel and showing it again reopens it at the dragged width — and is bounded at both ends: it can crush neither the panel nor the editing area to nothing (INV-056). |
| **Diagram Graph** | The structured, in-memory model of a node/arrow Mermaid Diagram that the Flowchart Builder edits: a Diagram Kind, a Flow Direction, an ordered list of Diagram Nodes, and an ordered list of Diagram Edges. It is the graphical counterpart of a Mermaid Diagram's source text — parsed from that source and emitted back to it as canonical Mermaid, so the source text stays the single source of truth (INV-051). A Diagram Graph models only *structure*: node **layout** — where each node sits — is Mermaid's to compute and is never part of the graph, so the Flowchart Builder's on-canvas positions are view state that never reaches the Markdown Document. |
| **Diagram Kind** | Which node/arrow diagram a Diagram Graph is — **Flowchart** today; other node/arrow kinds (state, ER, class) may follow. It selects the Mermaid header keyword and the node/edge syntax used to parse and emit the graph. Non-graph Mermaid diagrams (sequence, gantt, pie) are **not** Diagram Kinds: they are authored as text with a live Diagram Preview, not in the Flowchart Builder. |
| **Flow Direction** | The direction a Diagram Graph flows — one of **Top-Down**, **Left-Right**, **Bottom-Up**, or **Right-Left** — emitted as Mermaid's `TD`, `LR`, `BT`, or `RL`. |
| **Diagram Node** | One node of a Diagram Graph: a Node Id (its stable identifier in Mermaid source), a Node Label (the text shown in the node), and a Node Shape. |
| **Node Id** | The stable identifier that names a Diagram Node in Mermaid source and that a Diagram Edge references. Unique within a Diagram Graph and never empty (INV-052). An Id read from existing source is preserved verbatim; an Id the Flowchart Builder mints for a new node is a fresh `n1`, `n2`, … . |
| **Node Label** | The text shown inside a Diagram Node. May be empty, and may contain spaces and punctuation — it is quoted in the emitted source, so it survives a Round-Trip through the Mermaid source (INV-051). |
| **Node Shape** | The outline a Diagram Node is drawn with — **Rectangle**, **Rounded**, **Stadium**, **Diamond** (a decision), or **Circle** — emitted as the matching Mermaid node-shape delimiters. |
| **Diagram Edge** | A directed connection from one Diagram Node to another: a **From** Node Id, a **To** Node Id, an optional Edge Label, and an Edge Kind. Every Diagram Edge references Diagram Nodes the graph declares — an edge to a node that is not there cannot exist (INV-052). |
| **Edge Label** | The optional text shown on a Diagram Edge (e.g. `yes` / `no` on a decision's two branches). Absent when the edge carries no text. |
| **Edge Kind** | How a Diagram Edge is drawn — **Arrow** (`-->`), **Dotted** (`-.->`), **Thick** (`==>`), or **Open** (`---`, a line with no arrowhead). |
| **Flowchart Builder** | The graphical authoring surface for a Diagram Graph: a modal window, opened from the Command Bar, that presents the Mermaid Diagram at the caret as a **drag-and-drop canvas** — Diagram Nodes as draggable boxes joined by drawn Diagram Edges — beside a live Diagram Preview. The user adds and moves nodes, draws edges between them, renames and reshapes nodes, and sets the Flow Direction; **Insert** writes the resulting Diagram Graph back as the Mermaid Diagram's source (canonical Markdown), whereas editing within the builder and **Cancel** change no Markdown Document (INV-053). It is the graphical half of the Flow Charts feature, over the same Mermaid source that text authoring and the Diagram Preview already use — a new authoring surface, not a new render path. |
